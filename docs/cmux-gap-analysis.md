# Canvas vs cmux: UI/UX Gap Analysis

Актуально по состоянию на 25 марта 2026 года.

Этот документ опирается на:

1. текущий код и живое окно `canvas-app`;
2. живое окно установленного на машине `cmux`;
3. публичные страницы и документацию `cmux`.

## Короткий вывод

`canvas` уже интереснее `cmux` как продуктовая идея.
У нас сильнее тезис про spatial workdesk, layout lenses и работу с несколькими активными контекстами.

Но прямо сейчас `cmux` выглядит более зрелым продуктом по трём причинам:

1. Он content-first.
   Терминал получает почти всё внимание, а хром интерфейса минимален.
2. Он лучше маршрутизирует внимание.
   У него есть уведомления, unread-состояния и заметные сигналы от агентов.
3. Он лучше связан с автоматизацией.
   UI, CLI, сокет API и агентные хуки работают как одна система.

Если говорить жёстко: сегодня `canvas` уже богаче как эксперимент, но `cmux` воспринимается более собранным и менее "prototype-coded" в ежедневном использовании.

## Что у Canvas уже хорошо

1. Есть отчётливая продуктовая ось.
   Spatial desk, свободное перемещение, grid lens, split lens и `Expose` дают собственный характер, а не очередной GUI-терминал.

2. Уже есть настоящая многодесковая модель.
   `Workdesk` как сущность, несколько desk'ов, persist, duplicate/delete, focus state и layout mode уже делают продукт не игрушкой.

3. Терминал встроен как first-class surface.
   Это не мок UI поверх идеи, а живая PTY + `libghostty-vt` связка.

4. Горячие клавиши и жесты уже на хорошем уровне для прототипа.
   Есть remap shortcut'ов, trackpad pan/zoom, directional navigation, terminal selection, copy/paste.

## Где cmux сильнее прямо сейчас

### 1. Сильнее иерархия интерфейса

У `cmux` почти весь экран отдан под контент.
Левая колонка узкая, верхний chrome нативный, вторичные функции спрятаны в меню, настройки и шорткаты.

В `canvas` одновременно конкурируют:

1. левая rail;
2. нижний левый control dock;
3. нижний правый status dock;
4. chrome самих pane'ов;
5. служебные карточки вроде `Ghostty bridge`.

Из-за этого взгляд чаще считывает управление, чем саму рабочую поверхность.

### 2. Сильнее attention model

`cmux` строит UX вокруг сигнала "где агент или задача требует внимания":

1. notification rings;
2. unread badges;
3. notification panel;
4. jump to latest unread;
5. desktop notifications.

`canvas` умеет держать несколько pane'ов и workdesk'ов, но пока почти не помогает понять, куда смотреть в первую очередь.
Для agent-oriented продукта это критический пробел.

### 3. Сильнее автоматизация вокруг UI

У `cmux` UI не живёт отдельно от automation layer.
CLI и Unix socket позволяют:

1. создавать и выбирать workspace;
2. сплитить поверхности;
3. слать input;
4. поднимать уведомления;
5. писать sidebar status/progress/log;
6. управлять browser surface.

Это делает интерфейс не просто красивым, а интегрируемым в агентные workflow.

У `canvas` пока продуктовая автоматизация почти не экспонирована наружу.

### 4. Сильнее product polish

В `cmux` чувствуется сознательная экономия интерфейса:

1. меньше постоянного текста;
2. меньше диагностических меток;
3. меньше декоративных статусов;
4. более нативное ощущение окна;
5. меньше объяснений на экране, больше самопонятных affordance.

В `canvas` сейчас много хороших идей показаны слишком буквально.
Продукт объясняет себя через текст и статус-чипы вместо того, чтобы опираться на композицию и поведение.

## Что у Canvas потенциально лучше, чем у cmux

Если довести до ума, `canvas` может быть сильнее `cmux` в трёх местах:

1. Пространственная память, а не только список workspace'ов.
2. Переключение между свободным, grid и split режимами на одних и тех же сущностях.
3. Сценарии с несколькими агентами, где важны кластеры контекста, а не только вертикальные вкладки.

Это и есть наш unfair advantage.
Нельзя разменять его на копирование `cmux` один в один.

## Главные текущие UX-разрывы в Canvas

### P0. Слишком много постоянного chrome

Сейчас интерфейс перегружен низкоприоритетными элементами:

1. широкая левая rail;
2. отдельная карточка `Shortcuts`;
3. отдельная карточка `Ghostty bridge`;
4. большой нижний control dock;
5. большой нижний status dock;
6. насыщенный header каждого pane.

Следствие:
терминал и сам canvas получают меньше воздуха, чем должны.

### P0. Нет модели внимания для агентов

Нужны:

1. unread/attention state у pane;
2. unread/attention state у workdesk;
3. jump to next pane needing attention;
4. desktop/banner notifications;
5. история сигналов.

Без этого spatial layout плохо масштабируется при реальной параллельной работе.

### P1. Боковая rail пока слишком "описательная", а не операционная

В `canvas` rail больше объясняет концепцию, чем ускоряет действия.
В хорошей версии rail должна в первую очередь отвечать на вопросы:

1. какой desk живой;
2. где агент ждёт;
3. какой cwd и branch у активной сессии;
4. где идёт прогресс;
5. куда мне прыгнуть сейчас.

### P1. Слишком много debug-state попадает в основной UX

`ghostty linked`, vendor path, layout/mode/zoom chips полезны нам как разработчикам, но не должны быть постоянным слоем пользовательского интерфейса.

Это нужно или:

1. убирать в debug overlay;
2. показывать только в dev build;
3. прятать за inspector / command palette.

### P1. Недостаточно "entry points" в продукт

Нужны более сильные стартовые сценарии:

1. new shell desk;
2. agent review desk;
3. debug desk;
4. implementation desk;
5. reopen recent desk / template.

Сейчас пользователь получает мощную систему, но не получает готовый ритм входа в неё.

### P2. Нет внешнего product API для orchestration

Для agent-native инструмента нужен наружный интерфейс хотя бы уровня:

1. создать pane/workdesk;
2. сфокусировать pane/workdesk;
3. выставить статус;
4. выставить progress;
5. поднять notification;
6. записать activity log;
7. прочитать текущий desk state.

Это один из самых важных дифференциаторов `cmux`.

## Что стоит сделать первым

### Этап 1. Сделать интерфейс content-first

Это самый выгодный шаг.
Даже без новых фич визуальное качество резко вырастет.

Сделать:

1. сузить левую rail;
2. убрать `Ghostty bridge` из основного UI;
3. убрать или радикально сократить нижний правый status dock;
4. сделать нижний control dock компактнее;
5. облегчить header pane и сократить количество постоянных badge'ей;
6. увеличить визуальный вес terminal surface относительно chrome.

Цель:
при первом взгляде пользователь должен видеть работу, а не систему управления работой.

### Этап 2. Добавить attention model

Минимальный пакет:

1. `pane_attention_state = idle | working | waiting | error`;
2. бейджи на desk cards;
3. jump to next attention;
4. ephemeral toast/notification center;
5. API для агентных хуков.

Это даст `canvas` право быть реальным agent workspace, а не просто spatial terminal prototype.

### Этап 3. Сделать rail живой

В карточке desk должны появиться короткие, полезные сигналы:

1. cwd;
2. git branch;
3. last activity;
4. progress;
5. unread count;
6. current intent.

Важно:
не превращать rail в длинную карточную ленту.
Нужна очень плотная, операционная подача.

### Этап 4. Открыть product API

Даже если browser surface пока вне v0, API для desk/pane/status/notification уже нужен.

Если сделать только одну интеграционную вещь после attention model, это должен быть локальный socket/CLI слой.

### Этап 5. Потом решать вопрос browser surface

Встроенный браузер у `cmux` очень сильный differentiator, но для `canvas` это не первый ход.

Правильный порядок такой:

1. content-first shell polish;
2. attention model;
3. product API;
4. starter flows / templates;
5. browser surface.

Иначе мы добавим дорогую функцию раньше, чем соберём собственный характер продукта.

## Практический бар "не хуже cmux"

Чтобы можно было честно сказать, что `canvas` не хуже `cmux` по UI/UX, нам нужен минимум:

1. интерфейс, который визуально отдаёт приоритет контенту;
2. работающая модель внимания для агентных pane'ов;
3. компактная и полезная sidebar metadata;
4. быстрый jump/navigation между важными контекстами;
5. наружный API или hooks для automation;
6. более зрелое ощущение окна без явного debug chrome.

Без этих пунктов мы можем быть интереснее концептуально, но будем проигрывать в ежедневном использовании.

## Рекомендуемый следующий implementation slice

Если делать ближайший, самый выгодный кусок работы, я бы брал такой пакет:

1. dev-only diagnostic chrome;
2. compact sidebar;
3. compact pane header;
4. remove bottom-right status bar in favor of a smaller contextual strip;
5. workdesk rename;
6. pane/workdesk attention badges and next-attention shortcut.

Это уже сильно приблизит ощущение продукта к "daily-driver quality", не ломая spatial thesis.
